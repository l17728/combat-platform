# v2.5 Sprint 设计 — Hermes 通用检索工具集（Tool-using LLM）

- 文档版本：v2.5 draft（2026-05-31）
- 起点：master @ 307a3d8（v2.4.0）
- 旗舰主题：把 Hermes 从「intent 路由器」升级为「会自己查 schema/KG/数据的 Tool-using Agent」
- 配套扫尾：v2.4 5 专家 review 中遗留的 Schema-as-UI、UX 中期、fallback search 性能等

## 1. 背景与问题

当前 `apps/backend/src/hermes.ts` 是规则版意图路由：用正则识别问句 → 调对应的内部函数（`find-helpers`、`ticket-by-pb`、`owner`、`status`、`contribution-by-person`、`person-workload`、`recent-changes`），命不中则进入 `fallback-search`（全 nodeType 暴力 substring）。

`hermes-agent.ts` + `opencode-runner.ts` 已经把 opencode + glm-5 跑通，但实际只暴露了 `lookup` / `getContext` / `recommendHelpers` / `ticketTabs` 与 4 个 Welink 专用工具，**做不到 introspection 与聚合**：

- 「有多少员工」→ 没有 `count` 工具 → 落 `fallback-search` → 空。
- 「本月新增几条攻关单」→ 没有时间窗过滤 → 误命中或空。
- 「admin 改过哪些 schema」→ 没有 audit 工具 → 空。

用户原话：「工具少一个就加一个，是不是太被动？而是开发一个**检索的工具**，AI 可以自己去查 schema 或者知识图谱，自主找到答案。」

结论：把 Hermes 重做成**通用 Tool-using Agent**——LLM 看 schema 自决，不再硬编码 intent。

## 2. 工具集设计（核心）

放在 `apps/backend/src/hermes-tools.ts`（纯函数 + JSON Schema），由 `apps/backend/hermes-workspace/.opencode/tools/hermes.ts` 通过 HTTP 暴露给 opencode。所有工具**只读为默认**，写工具默认禁用、需显式开关。

| # | 工具 | 入参（JSON Schema 摘要） | 出参 | 实现路径 |
|---|------|---------------------------|------|---------|
| 1 | `list_node_types` | `{}` | `[{nodeType,label,fields:[{name,type,required,enumValues?}]}]` | `registry.getConfig().nodeTypes` |
| 2 | `describe_node_type` | `{nodeType:string}` | 单类型完整 schema + sample(1) | `registry` + `repo.queryNodes(nt,{limit:1})` |
| 3 | `count_nodes` | `{nodeType, filter?:{[k]:v\|{op,val}}}` | `{count:number}` | 新增 `repo.countNodes` 走 `COUNT(*) WHERE node_type=? AND json_extract(...)` |
| 4 | `query_nodes` | `{nodeType, filter?, limit?:<=50, offset?, sort?:{field,dir}}` | `[{id,nodeType,properties,updatedAt}]` | `repo.queryNodes` + 字段索引 |
| 5 | `get_node` | `{id:string}` | `{node,progress(last 5),related:{in,out}}` | 复用 `GET /query/context/:id` |
| 6 | `search_text` | `{q:string, scope?:string[], limit?:<=20}` | `[{id,nodeType,summary,score}]` | 复用 `GET /query/search`，scope 限制 nodeType |
| 7 | `traverse_graph` | `{startId, edgeTypes?:string[], depth?:1-3}` | `{nodes,edges}` 限 200 节点 | `repo.listEdges` BFS |
| 8 | `get_progress` | `{nodeId, limit?:<=20}` | `[ProgressLog]` | `repo.listProgress` |
| 9 | `get_audit` | `{entityId?, actor?, action?, since?, until?, limit?<=50}` | `[AuditEntry]` | `repo.listAuditLog` |
| 10 | `aggregate` | `{nodeType, groupBy:string, agg?:'count'\|'sum:field'\|'avg:field', filter?, having?}` | `[{key,value}]` | 受限 SQL：白名单 agg/groupBy，禁通用 SQL |
| 11 | `dashboard_metric` | `{key:'活跃攻关单'\|'本周新增'\|'人员负载'\|...}` | 预聚合指标 | 复用 `GET /api/dashboard` |
| 12 | `recommend_helpers` | `{ticketId}` | 现有结构 | 已有 |
| 13 | `ticket_tabs` | `{ticketId}` | 现有结构 | 已有 |
| 14 | `welink_search` / `welink_timeline` / `welink_gap_analysis` | 现有 | 现有 | 已有，纳入统一文档 |

`filter` 统一语法（避免 LLM 走 SQL）：

```json
{
  "状态": "处理中",
  "updatedAt": { "op": "gte", "val": "2026-05-01" },
  "当前处理人": { "op": "in", "val": ["张三","李四"] }
}
```

支持 op：`eq, ne, gt, gte, lt, lte, in, contains, exists`。后端把 op 翻译成参数化 SQL（`json_extract` + 索引），任何不支持的 op 直接 422，**绝不拼字符串**。

工具产物体积上限：单工具响应 ≤ 32 KB，超出截断并附 `truncated:true` 提示，让 LLM 继续翻页。

## 3. 工具调用编排

### 3.1 协议
采用 **OpenAI-compatible tool-calling**（opencode + glm-5 已原生支持），不走 ReAct 自由文本——后者 JSON 提取脆弱、易 hallucinate 字段名。每工具用 `@opencode-ai/plugin` 的 `tool({args: schema, execute})` 声明，参数 JSON Schema 由后端权威生成。

### 3.2 多轮
开启多轮，硬上限 `MAX_TOOL_HOPS=6`（env 可调）。退出条件：LLM 输出 final text（无 tool_calls）/ 触顶 / 全局超时 `HERMES_TIMEOUT_MS=180000`（已有）。每轮把 tool result 拼回上下文。

### 3.3 上下文预算
- 系统提示包含「数据字典摘要」(只列 nodeType+label+主键字段，不全量 dump fields)
- 工具返回若 >32KB 自动截断 → 写 `truncated_at` 提示
- 多轮历史保留**最近 3 轮**完整 tool_call/result，更早的折叠成 `<previous tool calls: N>` 摘要

### 3.4 失败 / 回退
- 工具调用 `try/catch`：业务 4xx 转 `{error,hint}` 给 LLM 重试；5xx/超时直接抛
- 顶层失败 → 回落到现有 `answerQuestion()` 规则引擎（已实现 `hermes.ask.agent_fallback` 日志）
- LLM 始终找不到答案（终止时未产出 final text）→ 同样 fallback 规则引擎

## 4. 安全 / 权限

| 维度 | 策略 |
|------|------|
| 身份 | 沿用现有 `signServiceToken()` 服务账号 token，工具内部按 LLM 暴露的 `asUser` 字段做二次校验（先按 admin 取数据，再过滤） |
| 私单 | 所有 attackTicket 工具结果都走 `filterAccessibleTickets(userId, ...)` 过滤，**默认隐藏私单**，除非调用方 token 是攻关单成员 |
| 写工具 | v2.5 **默认禁用**写工具（`create_node` / `update_node` / `welink_addMembers`），由 env `HERMES_WRITE_TOOLS=1` 显式打开；打开时强制要求二次确认（LLM 输出 `WRITE_CONFIRM: yes` 才执行） |
| 速率 | 单用户 60 次/分钟 工具调用上限，触顶 429 |
| 审计 | 新增 `hermes.tool.invoke` 结构化日志事件：`{user, tool, args, durationMs, ok}`；命中写工具的额外写入 `audit_log`（`action='HERMES_INVOKE'`） |
| 滥用检测 | 阈值告警：单会话 ≥ MAX_TOOL_HOPS 且最终无 final text → 日志 `hermes.tool.exhausted`，dashboard 暴露 |

## 7. 时间线 / 风险

预计 **5-7 工作日**（4 桶并行计入 buffer）。

| 风险 | 缓解 |
|------|------|
| opencode prompt + 工具 trace 把 context 撑爆，glm-5 截断 | 工具结果 32KB 上限 + 历史折叠摘要 + `MAX_TOOL_HOPS=6` |
| LLM hallucinate 字段名 / nodeType | 系统提示先调 `list_node_types`；工具入参 JSON Schema 严校，未知 nodeType 直接 422 |
| 工具串太多轮 latency 飙升 | per-tool budget 8s + 全局 180s；P95 监控 dashboard 指标 |
| 与现有规则 fallback 抢答 | 双路并存：`HERMES_AGENT=on` 时优先 agent，失败/超时静默回落规则；CLI `hermes:ask --engine=rule|agent` 强制单路用于回归 |
| 写工具误触发 | 默认禁用；启用需 `HERMES_WRITE_TOOLS=1` + LLM 输出 `WRITE_CONFIRM: yes`；写动作全部 audit |
| 私单泄漏 | `filterAccessibleTickets` 收口在工具层一次，绝不让工具直接返回原始 query 结果 |
