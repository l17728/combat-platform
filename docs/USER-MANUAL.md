## 1. 快速上手

### 1.1 进入平台

- 默认地址：`http://<服务器IP>:5173`（开发模式）或 `http://<服务器IP>/`（部署模式）
- 后端 API：`http://<服务器IP>:3001`

### 1.2 首页大盘

打开首页（`/`）即可看到：

- **攻关单总数 / 进行中 / 已闭环**：核心计数
- **贡献总数 / 待审批提议**：荣誉与实体解析队列规模
- **状态分布**：所有攻关单按 `状态` 字段分布
- **Top 贡献人**：贡献数最高的 5 人
- **冲突 / 重叠**：本平台检出的"同负责人多并发"或"同问题单号重叠"情况
- **今日动态**：今天的进展条数 + 涉及的攻关单数
- **最近活跃攻关单**：按最后变更时间倒序的 5 条
- **下方功能模块卡片**：所有功能的快捷入口

### 1.3 顶部导航与角色切换

顶栏菜单是主导航：「首页 / 攻关作战台 / 作战表 / 荣誉殿堂 / 贡献录入 / 导入 / ...」。

顶栏最右侧有「**角色:普通 / Leader / 管理员**」下拉菜单 — 切换后会立即影响你的权限边界（详见 [§2 角色与权限](#2-角色与权限)）。

## 2. 角色与权限

平台有 3 个角色：

| 角色 | 显示名 | 主要能力 |
|---|---|---|
| `normal` | 普通 | 浏览、创建/编辑/状态流转/贡献录入（**不可标定贡献等级**）|
| `leader` | Leader | 普通的全部 + **可标定贡献等级**（普通/关键/核心）+ SLA 配置 + 提议审批 |
| `admin` | 管理员 | 当前与 Leader 等价（保留扩展位） |

**唯一受角色限制的写操作**：在「贡献录入」中填写 `贡献等级` 字段。普通成员若提交带 `贡献等级` 的请求，会被后端拒绝并提示 "仅 Leader 可标定贡献等级"。

> 其他写操作（攻关单 CRUD、进展、状态流转、Schema 变更、合并等）**没有**角色门控，所有人都能操作——一切以审计日志为准。这是一个轻量级团队工具，目的是协作而非门禁。

切换角色后，浏览器会在 localStorage 记住你的角色（key = `combat-role`），下次刷新仍然生效。

## 3. 攻关工作流

### 3.1 场景 A：我有一个新的攻关单要登记

**入口**：顶栏「攻关作战台」→ 点击「**新增行**」。

**步骤**：
1. 至少填写**必填字段**（红色 `*`）：标题 + 状态。
2. 可选但**强烈推荐**填写：问题单号（PB-xxx）、事件级别（P1/P2/P3）、当前处理人、攻关组长。
3. 点击「创建」。

> 自动联动：保存后台后，系统会立即在后台异步执行 (a) 冲突扫描 (b) SLA 上升扫描 (c) 跟催扫描，无需手动触发；超期/冲突会出现在「冲突」和「跟催提醒」中。

**字段语义速查**：

| 字段 | 含义 / 建议填法 |
|---|---|
| 攻关单号 | 系统不强制，可留空；用作业务参考号 |
| 标题 | 必填；简洁描述（如「华南-XXX 客户 ModelArts 训练任务卡死」） |
| 状态 | 必填，五选一：`待响应 / 处理中 / 进行中 / 已解决 / 已关闭` |
| 事件级别 | P1/P2/P3/P4A — **决定 SLA 时长**（详见 §6.1） |
| 问题单号 | PB-xxx 格式，**跨视图关联的核心 anchor** — 与 P3 事件单、贡献等共享 |
| 当前处理人 | 输入姓名，系统会自动建立到 `person` 节点的关系（自动创建人员节点） |
| 客户要求解决时间 | 接近 deadline 时（3 天内）会自动触发跟催提醒 |
| 是否需CCB | 选「是」会生成 CCB 提醒 |

### 3.2 场景 B：我要在攻关单上同步进展

**入口**：顶栏「攻关作战台」→ 点击某行的「标题」打开详情页。

**步骤**：
1. 切到「**进展同步**」Tab。
2. 在文本框输入这次的进展（例如「已联系厂商定位，初步怀疑驱动版本问题，明早 9 点继续」）。
3. 点击「**追加进展**」。

> 进展是**只追加不修改**的时间线 — 每条进展自动带序号 `#1 / #2 / ...` 和当时的状态快照。**永远不要靠"改个字段"来表达进展，会丢失历史。**

### 3.3 场景 C：我要把攻关单"转给下一个状态"

例如：从 `待响应` → `处理中`，或从 `处理中` → `已解决`。

**入口**：「攻关作战台」→ 点击「标题」打开详情页 → 「基础信息」Tab → 滚到「**状态流转**」区块。

**步骤**：
1. 在「目标状态」下拉中选择目的状态（`待响应 / 处理中 / 进行中 / 已解决 / 已关闭`）。
2. 在「备注」输入一句话说明原因（可选但**强烈建议填**，例如「客户反馈业务恢复，等观察期」）。
3. 点击「**流转**」。

> 系统在一次原子操作里完成两件事：(a) 更新状态字段 (b) 追加一条进展 `状态变更：处理中→已解决；客户反馈业务恢复，等观察期`。永远可追溯。

### 3.4 场景 D：我想找一个合适的攻关协助者

**入口**：攻关单详情页 → 「基础信息」Tab → 滚到「**找帮手**」区块。

系统会自动列出推荐人选（最多 5 人），含 **加权得分** 与 **推荐理由**，例如：

```
张三（分数 11）：曾处理共享问题单「PB-12345」的攻关单「网络拓扑异常」；
                  在共享问题单「PB-12345」相关贡献「定位根因」（核心）

李四（分数 3）：曾处理共享问题单「PB-12345」的攻关单「容器 OOM」
```

**推荐规则**（自动）：
- 在**同一个问题单号**下处理过其他攻关单的人 → +3 分
- 在同一个问题单号下做过相关贡献 → +1/+2/+3（普通/关键/核心）
- 当前处理人本人不会被推荐

也可以用 Hermes 自然语言查询：「PB-12345 找谁帮忙」、「找帮手 模型训练卡死」。

### 3.5 场景 E：我要给攻关单写一条结构化日报

**入口**：攻关单详情页 → 「**日报更新**」Tab → 点击「创建」按钮。

**步骤**：
1. 选「日报类型」：`进展通报` 或 `风险通报`。
2. 「**当前进展**」（必填，多行文本）：阶段性结论 + 待解决问题。
3. 「下一步计划」（选填）：明天/下周做什么。
4. 提交后默认是「**草稿**」状态 — 主动点「发布」才会标记为 `已发布`。

> 日报条目是独立于进展（ProgressLog）的一种**正式通报**机制。两者并存：进展是技术细节流水，日报是给 leader / 客户看的结构化通报。

### 3.6 场景 F：浏览某攻关单的"关联全景"

每个攻关单详情页右上角有「**关联全景**」链接，点进去看到：

- **出边 / 入边**：直接关联的 person、问题单 anchor、其他攻关单
- **共同锚点（coAnchored）**：与本单共享同一 `问题单号` / `事件单号` 的其他记录（贡献、P3 事件等）
- **冲突 / 重叠**：本单参与的 `CONFLICTS_WITH` / `OVERLAPS_WITH` 关系
- **手工关联线（manualLinks）**：人为标注的关联（带备注）
- **多跳展开（expanded）**：通过 URL 参数 `?depth=2..5` 查看 2~5 跳之内的所有间接关联

「关联全景」是平台**最重要的认知工具**。当问题复杂时，先打开全景看一眼，比读 20 个 Excel 都有效。

## 4. 贡献录入与荣誉殿堂

### 4.1 场景 A：录入一笔贡献

**入口**：顶栏「**贡献录入**」→ 「新增行」。

**字段**：

| 字段 | 必填？ | 说明 |
|---|---|---|
| 贡献人 | 是 | 姓名；自动关联到 person 节点 |
| 贡献类型 | 是 | `发现 / 设计 / 实施 / 协调 / 公关` |
| **贡献等级** | 否 | `普通 / 关键 / 核心` — **仅 Leader 可填**！普通成员留空即可，由 Leader 后续审定 |
| 贡献描述 | 否 | 一句话说明做了什么 |
| 关联攻关单 | 否 | 攻关单号 或 标题 — **强烈建议填**，否则不会在荣誉殿堂"个人页"出现关联记录 |
| 关联问题单 | 否 | PB-xxx — 与攻关单共享问题单的 anchor |
| 周期 | 否 | 自定义（如 `2026-Q2`、`2026-05`），便于排行榜按周期过滤 |
| 记录时间 / 记录人 | 否 | 通常自动填 |

**普通成员的工作流**：先创建一条贡献记录（贡献等级留空），由 Leader 在后续 review 时在「贡献录入」里点击「编辑」加上贡献等级。

### 4.2 场景 B：查看荣誉排行榜

**入口**：顶栏「**荣誉殿堂**」。

显示：「名次 / 贡献人 / 加权得分 / 贡献数 / 各等级分布」。

**加权规则**：核心 = 8 分 / 关键 = 3 分 / 普通 = 1 分；未标等级按 1 分计。

可在搜索框输入 `周期` 字符串过滤（如 `2026-Q2`），只看当周期排名。

### 4.3 场景 C：查看某个人的全部贡献

**入口**：「荣誉殿堂」中点击姓名 → 「个人档案」。或 URL `/honor/<姓名>`。

展示这个人的所有贡献记录 + 每条贡献关联的攻关单链接，可一键跳转回攻关单详情。

### 4.4 场景 D：Leader 标定贡献等级

1. 切到「Leader」角色（顶栏右上角）。
2. 进入「贡献录入」，找到普通成员录入的待标定记录。
3. 点击「编辑」→ 在「贡献等级」选择「核心 / 关键 / 普通」→「保存」。

> 普通成员尝试同样操作时，系统会拒绝并提示 "仅 Leader 可标定贡献等级"。

## 5. 跨视图协作

### 5.1 场景 A：在攻关单之间发现"同负责人多并发"

**入口**：顶栏「**冲突**」（红色高亮）。

系统会自动列出冲突对，例如：

| 类型 | 攻关单 A | 攻关单 B | 原因 |
|---|---|---|---|
| CONFLICTS_WITH | 标题1 | 标题2 | 同负责人多并发：张三 |
| OVERLAPS_WITH | 标题3 | 标题4 | 同问题单：PB-12345 |

- **同负责人多并发**：当一个人作为多个**活跃**攻关单（状态在 `待响应/处理中/进行中`）的 `当前处理人` 时，每两两关系都会被标记
- **同问题单重叠**：当多个攻关单（任意状态）填了同一个 `问题单号`，每两两关系都会被标记

这些边是**派生**的（自动同步，不可手动编辑）；如发现关闭后未刷新，可在该页点「**重新扫描**」按钮强制重建。

### 5.2 场景 B：手动标注两条记录的关联

有时业务关系无法通过字段表达（例如："这个攻关单和这个变更是相关的"）。

**入口**：进入任一记录的「关联全景」页 → 在「手工关联」区块用 source/target 选择器 → 写一句 `备注`（说明关系）→ 提交。

后端会创建一条 `RELATES_TO` 边，双方的「关联全景」都会显示。这是 schema 之外的**自由形式标注**。

### 5.3 场景 C：把同一个人的多条记录合并

**问题**：导入时由于姓名/工号不同，"张三" 在系统里有两个 person 节点。

**入口**：顶栏「**人员合并**」。

**步骤**：
1. 选择 `from`（要被合并掉的）和 `to`（保留的）两个 person 节点。
2. 点「**预览**」：系统显示将合并哪些字段、迁移多少条边。
3. 确认无误后点「**执行合并**」。

> **不可逆操作**！请先预览。系统会：
> - 把 `from` 的字段并入 `to`（`to` 已有的优先）
> - 把 `from` 的所有关联边重新指向 `to`（去重）
> - 删除 `from` 节点
> - 写 audit_log（`action=MERGE`）

### 5.4 场景 D：审批系统提议的"可能是同一个人"

**入口**：顶栏「**关系审批**」。

系统会自动扫描可能同人的候选（`SAME_AS` 提议），列出来等审批。

| 状态 | 含义 |
|---|---|
| 待审批 | 待人工决策 |
| 已通过 | 触发实际合并 |
| 已拒绝 | 后续扫描不会再生成此条 |

操作：「**通过**」/「**拒绝**」/「**修正**」（修正后可指定合并到另一个人）。

## 6. 自动化与提醒

### 6.1 场景 A：SLA 自动上升

平台预置 SLA 规则：

| 事件级别 | SLA 时长 | 上升角色 |
|---|---|---|
| P1 | 2 小时 | 运维Leader |
| P2 | 8 小时 | 运维Leader |
| P3 | 24 小时 | 值班接口人 |
| P4A | 4 小时 | 值班接口人 |

每 5 分钟系统自动扫一遍；超时的活跃攻关单会：
- 在审计日志写一条 `ESCALATE` 事件
- 创建一条 `ESCALATED_TO` 关联（指向当前处理人）
- 标记为「已上升」，不会重复触发

**入口（查看 / 修改 SLA）**：顶栏「**SLA上升**」。Leader 可修改规则（事件级别 / slaHours / 上升角色三元组）。

**入口（看哪些单上升了）**：顶栏「审计日志」过滤 `action=ESCALATE`，或在攻关单关联全景里看 `ESCALATED_TO` 边。

### 6.2 场景 B：管理跟催提醒队列

平台自动检测三类跟催场景并生成提醒：

| 类型 | 触发条件 |
|---|---|
| 问题单跟催 | 攻关单**未关闭**且**最近一次进展已超过 3 天** |
| FE Deadline 提醒 | 攻关单填了「客户要求解决时间」且**距 deadline ≤ 3 天** |
| CCB 提醒 | 攻关单填了「是否需CCB=是」 |

7 天内同一条（kind + ticket + 收件人）不会重复生成。

**入口**：顶栏「**跟催提醒**」。

操作（每条提醒一行）：
- **发送**：通过当前邮件配置发送邮件
- **忽略**：标记为已忽略，不再骚扰

> 邮件发送需要先在「邮件」页配置 SMTP（host/port/username/password/from）。否则只是 stub（控制台日志，不实际外发）。

### 6.3 场景 C：手动触发一次全部后台扫描

「跟催提醒 / SLA上升 / 冲突 / 提议」每个都有「**重新扫描**」按钮可单独触发，也可以一次跑全部：

- 顶栏「**冲突**」页 → 「扫描」按钮
- 或使用 CLI：`npm run cli -- jobs:tick`

会返回 `{conflicts, overlaps, escalated, reminders, proposals}` 的产出数量。

### 6.4 场景 D：查看当前值班人

**入口**：顶栏「**Oncall**」可查看所有值班记录；通过 CLI 查询当前生效值班人：

```bash
npm run cli -- oncall:current --domain 运维
```

值班是按日期派生的：今天的日期落在某条记录的 `[起, 止]` 之间，该记录就生效。

## 7. Hermes 智能问答

**入口**：顶栏「**Hermes 问答**」。

Hermes 是一个**基于规则的中文问答引擎**（无需 LLM 凭据），会按你的问题自动识别意图，返回中文答案 + 引用的节点列表 + 可视化组件（表格/卡片/统计/时间线/Mermaid）。

### 7.1 八种意图举例

| 你问的话 | 系统识别为 | 返回什么 |
|---|---|---|
| `PB-12345 找谁帮忙` | find-helpers | 找帮手推荐 Top 5（卡片网格） |
| `PB-12345` 或 `问题单号: PB-12345` | ticket-by-pb | 该问题单下所有攻关单（表格） |
| `谁负责 模型训练卡死` | owner | 标题匹配的攻关单 + 当前处理人（表格） |
| `状态 模型训练卡死` 或 `怎么样 PB-12345` | status | 攻关单状态 + 最近一条进展（卡片） |
| `张三 贡献了什么` | contribution-by-person | 该人最近 5 条贡献（表格） |
| `谁最忙` / `负载最重` / `活跃单最多` | person-workload | 人员活跃单数排名 Top 5（统计图） |
| `今天 / 本周 / 最近 谁动了什么` | recent-changes | 当窗口内变动的攻关单（表格） |
| `(其他)` | fallback-search | 全字段全文检索 |

### 7.2 把常用问题"固定"在侧边栏

在 Hermes 回答下方点「**固定**」按钮，给它起个名字（如「P1 高危单」），就会出现在 Hermes 页的左侧 Pinned 区域，下次点击直接看结果。

### 7.3 通过 CLI 用 Hermes（适合 agent）

```bash
npm run cli -- hermes:ask "PB-12345 找谁帮忙"
```

返回完整 JSON：`{question, intent, answer, citations[], uiSpec?}`。

## 8. 数据导入 / 导出

### 8.1 场景 A：从 Excel 导入数据

**入口**：顶栏「**导入**」。

**步骤**：
1. 选择目标 nodeType（如 `attackTicket / contribution / oncall ...`）
2. 上传 `.xlsx` 文件
3. 先勾选 「**仅预览（Dry Run）**」 → 系统会告诉你「将创建 N 条、更新 M 条、跳过 K 条」并列出每行结果
4. 确认无误，取消勾选「仅预览」→ 重新提交即真实导入

> 列名匹配规则：先按 Schema 中字段的 `name / label / aliases` 匹配 Excel 列头。例如 `attackTicket.标题` 的 `aliases = ["title", "问题标题", "事件标题"]`，所以 Excel 列叫 `问题标题` 也能识别。

**特殊处理**：
- **去重**：每个 nodeType 的 `identityKeys` 决定按什么字段判重（如 attackTicket 按 `攻关单号` 判重）；重复行会**更新**，新行会**创建**
- **人员自动建**：导入攻关单时，"攻关申请人 / 当前处理人 / 攻关组长" 等字段的人名，会自动建立到 person 节点的关联（如果 person 不存在则创建）
- **错误行不阻断**：某行 schema 校验失败时跳过该行（在结果里看到 `skipped` 计数 + 跳过原因），其他正常导入

### 8.2 场景 B：导出某 nodeType 的 Excel

**入口**：任何作战表（EntityTable）页面 → 右上角「**导出 Excel**」按钮。

下载的文件命名为 `<nodeType>-<时间戳>.xlsx`，列名使用 schema 的 `label`（中文显示名）。

### 8.3 增强能力（已并行开发，即将上线）

- 多 Sheet 导入（按 nodeType 自动分发）
- 批量按 Identity 删除
- 自定义模板下载

## 9. Schema / 表结构管理

平台是 **配置驱动** 的 —— 加字段不需要数据库迁移。

### 9.1 场景 A：在某个作战表里加一个字段

**入口**：任何作战表（如「攻关作战台」）→ 表头右侧「**+字段**」按钮。

**步骤**：
1. 填字段名（英文/中文均可，作为存储键）
2. 填显示名（label，UI 列标题）
3. 选类型：`string / number / date / datetime / enum`
4. 点「添加」

字段**立即生效**，已有数据自动支持新字段（值为空），所有人页面刷新即可看到新列。

### 9.2 场景 B：给字段加别名（让 Excel 列名容错）

**入口**：表头上某字段的「**别名**」按钮 → 在弹窗里每行一个别名（如 `title`、`问题标题`）。

别名只影响**导入时的列匹配**，不影响存储。

### 9.3 场景 C：把某字段"退休"（隐藏但不丢数据）

**入口**：表头字段操作 → 「**退休**」按钮。

退休字段：
- 在表格/卡片中**不再展示**
- 导出 Excel 时**不包含**
- 但数据库里还在，下次「unretire」即可恢复

### 9.4 场景 D：从头新建一张作战表

**入口**：顶栏「**表结构管理**」（SchemaWizard）。

**步骤**：
1. 填 `nodeType`（英文 camelCase，需符合 `^[a-zA-Z][a-zA-Z0-9]*$`）
2. 填 `label`（中文显示名）
3. 在「字段编辑器」中逐字段配置（name / label / type / 是否必填 / enum 值 / refType / concept / anchor）
4. 提交后系统会自动：写 JSON 文件到 `config/schemas/<nodeType>.json` + 热加载（无需重启服务）

提示：在添加字段前，先在搜索框输入关键词（如「负责人」）→ 系统会列出所有现有表中的相关字段，**鼓励复用**（同样的 concept 可让跨视图查询/推荐生效）。

### 9.5 字段语义提示

| 字段属性 | 作用 |
|---|---|
| `required: true` | 必填校验（创建/更新时） |
| `enumValues: [...]` | enum 类型字段的合法值（如 状态字段的五种状态） |
| `refType: "person"` | 引用类型：填值后自动建立到目标 nodeType 节点的边 |
| `concept: "负责人"` | 跨视图复用的**语义标签**（不同字段 / 不同表填同样 concept 可表示"职责相同"） |
| `anchor: "问题单号"` | **锚点**：填值后会创建/复用一个 `问题单号` 类型的锚节点，并把当前记录关联过去 — 是跨视图发现的基石 |
| `aliases: [...]` | Excel 导入时的列名容错 |
| `retired: true` | 退休状态（UI 隐藏，数据库保留） |

## 10. 审计与合并

### 10.1 查看变更历史

**入口**：顶栏「**审计日志**」。

每一条记录形如：
```
2026-05-22T15:00:00Z   UPDATE   node   abc-123   by 张三   changes={状态:"处理中"}
```

可按以下维度过滤：`action`（CREATE/UPDATE/DELETE/MERGE/ESCALATE/...）、`entityType`（node/edge/schema/setting）、`entityId`（精确定位某个对象的所有历史）。

也可在每个攻关单详情页的「**历史记录**」Tab 看本单的审计流。

### 10.2 攻关单详情页的"历史记录" Tab

每个攻关单详情页提供专属审计视图，按时间线倒序展示该单的全部变更（创建 / 字段更新 / 状态流转 / 上升 / 日报发布 等）。

### 10.3 合并的不可逆性

如 §5.3 所述，人员合并不可撤销。建议工作流：

1. 切到 admin/leader 角色
2. 顶栏「人员合并」→ 选好 from / to
3. **务必**先点「预览」看清楚要并多少字段、多少边
4. 确认后才执行
5. 合并后审计日志里 `action=MERGE` 是这次操作的留痕

## 13. 常见问题排查

### 13.1 "我新加了字段，但 EntityTable 不显示"

- 确认浏览器**强制刷新**（Ctrl+Shift+R）
- 后端检查：`GET /api/schema/<nodeType>` 是否返回新字段；若没有 → 调用 `POST /api/schema/scan` 重新扫描配置目录

### 13.2 "录贡献时提示 '仅 Leader 可标定贡献等级'"

- 顶栏右上角切到「Leader」或「管理员」角色
- 或：先不填贡献等级提交，让 Leader 后续标定

### 13.3 "Excel 导入大量数据被跳过"

- 看导入结果里每个跳过行的 `reason`：通常是「字段「状态」必填」或「字段「状态」取值非法: xxx」
- 排查方向：
  - 必填字段是否所有行都填了？
  - enum 字段的值是否完全匹配（中文要一模一样，"完成" ≠ "已解决"）？
  - 列名是否能被识别？看 schema 里有没有这个 alias，必要时给字段加别名再导入

### 13.4 "已经过了 SLA 时间，但没有触发上升"

- SLA 扫描每 5 分钟一次。可以在 SLA 上升页强制扫描，或 CLI: `npm run cli -- escalation:scan`
- 检查 SLA 配置（顶栏「SLA上升」）：你的事件级别在规则表里吗？
- 是否已经上升过？同一个 ticket 不会重复上升 — 看 audit_log（`action=ESCALATE`）

### 13.5 "Hermes 找不到我想要的攻关单"

- 表述偏差：把"那个 PB-12345 单"改成具体的关键词（PB 号 / 标题片段 / 负责人姓名）
- 检查 ticket 的字段是否已填（如果标题为空，Hermes 找不到）
- fallback 是全字段全文检索 — 任意属性值都会被扫描，关键词太短可能噪声多

### 13.6 "我把人 A 合并到 B 了，但旧记录还在"

- 合并是**就地**操作：A 的字段并入 B，A 的所有边重新指向 B，A 节点被删除
- 如果旧 Excel 导入又生成了 A：查 audit_log 找回 A 的属性 → 重新执行合并
- 想避免：导入前在 person 表先 dedup

### 13.7 "邮件发不出去 / 测试邮件失败"

- 顶栏「邮件」页：先配置 SMTP（host / port / 用户名 / 密码 / from）
- 用「**测试邮件**」按钮验证：`POST /api/email/test` 会用当前配置发一封到指定地址，错误会显示完整 stack
- 网络隔离环境下需要确认服务器能 outbound 到 SMTP host

### 13.8 "KG 关联看起来"乱套"了"

可能原因：手动修改了 SQLite 文件，或 schema 字段定义改动后未重建派生边。

修复：在「关联全景」页或通过 CLI:
```bash
npm run cli -- kg:rebuild
```

系统会：
1. 删除所有派生边（REF / ANCHORED_TO / CONFLICTS_WITH / OVERLAPS_WITH）
2. 遍历所有节点，重新 syncRefEdges / syncAnchorEdges
3. 重新跑冲突扫描
4. 回收孤儿 anchor 节点

10 秒内即可完成（中等规模数据）。
